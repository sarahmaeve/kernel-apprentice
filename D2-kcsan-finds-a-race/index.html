$ D2 ./kcsan-finds-a-race

no crash · no error · just wrong numbers

Two workers process 200,000 events each and count them in a shared total. Every event is counted exactly once — and the total still comes up short, differently every run. That's a data race: the bug class that never crashes, just quietly corrupts. KCSAN catches it in the act and shows both sides.

make check LESSON=D2-kcsan-finds-a-race
Working in the workbench. Needs the debug kernel (make debug-kernel, one-time, shared with D1). Edit module/d2_race.c, then make check from the host.

./READ THE REPORT

BUG: KCSAN: data-race in d2_count_event [d2_race] / d2_count_event [d2_race]

write to 0xffffffffc03246c0 of 8 bytes by task 70 on cpu 1:
 d2_count_event+0x29/0x40 [d2_race]
 d2_worker+0x29/0x40 [d2_race]
 ...
read to 0xffffffffc03246c0 of 8 bytes by task 69 on cpu 0:
 d2_count_event+0x12/0x40 [d2_race]
 d2_worker+0x29/0x40 [d2_race]
 ...
kernel-apprentice: d2_race counted 347162 of 400000 events
  • Two stacks, two CPUs, one address — the definition of a data race: concurrent access, at least one write, no ordering. KCSAN plants a watchpoint on a sampled access and stalls; whoever touches the address during the stall is caught with it.
  • It's the same line, caught mid-split — the read at +0x12 and the write at +0x29 are d2_total++ torn into its load and store, photographed mid-interleave. This run lost 52,838 events; the next loses a different number.
  • Why the count is short — d2_total++ is a read-modify-write: both CPUs load N, both store N+1 — two events, one tick. Across 400,000 increments the total drifts low by a different amount every run.
  • Why it's miserable in production — production kernels don't run KCSAN; you just get metrics that don't add up, worse under load. The fix is trivial once you believe the diagnosis.
[02]

./FIND + FIX

Make the increment a single indivisible operation — the kernel's tool for a cross-CPU counter is atomic_long_t and friends. Note the check is not satisfiable by silencing KCSAN (e.g. data_race()): it independently requires the count to be exact.

PASS when the log shows counted 400000 of 400000 events and no BUG: KCSAN: data-race.

[03]

./GRADUATED HINTS

where is the race?

The report names d2_count_event twice — both racing parties are the same line, d2_total++, run concurrently by the two workers.

the right primitive?

A plain long can't be incremented safely from two CPUs. atomic_long_t / atomic_long_inc / atomic_long_read make the read-modify-write indivisible — and KCSAN treats marked atomics as intentional.

the fix 
static atomic_long_t d2_total = ATOMIC_LONG_INIT(0);

static noinline void d2_count_event(void)
{
	atomic_long_inc(&d2_total);
}

…and print it with atomic_long_read(&d2_total) in d2_init.
