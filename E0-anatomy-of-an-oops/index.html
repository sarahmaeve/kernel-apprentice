WARN · BUG · oops — read every field

The kernel doesn't fail silently — it reports, in a fixed format it has used for decades. A specimen module fails on purpose, three ways, so you can read each shape: a WARN ("shouldn't happen" — execution continues), a BUG() ("can't continue" — the writer is killed), and a NULL-deref oops (a page fault nobody planned). The kernel survives all three; a panic is the fourth rung, where nothing does.

make check LESSON=E0-anatomy-of-an-oops

The check writes warn, bug, oops into /proc/ka-anatomy (the fatal two from child processes), dumps all three reports, and prints the taint bitmask climbing: 4096 (O: out-of-tree module) → +512 (W: warned) → +128 (D: died).

./READ — the oops, field by field

BUG: kernel NULL pointer dereference, address: 0000000000000000
#PF: supervisor write access in kernel mode
#PF: error_code(0x0002) - not-present page
Oops: Oops: 0002 [#2] SMP NOPTI
CPU: 0 UID: 0 PID: 72 Comm: busybox Tainted: G      D W  O    6.18.33 #8 PREEMPT(voluntary)
Tainted: [D]=DIE, [W]=WARN, [O]=OOT_MODULE
RIP: 0010:e0_write.cold+0x16/0x3f [e0_anatomy]
Code: ... 48 8b 04 24 <c7> 00 ad de 00 00 e9 7c ff ff ff ...
RAX: 0000000000000000 RBX: 0000000000000005 RCX: 0000000000000000
Call Trace:
 <TASK>
 proc_reg_write … vfs_write … ksys_write … do_syscall_64
 </TASK>
Modules linked in: e0_anatomy(O)
---[ end trace 0000000000000000 ]---
  • headline — what kind of fault + the address that triggered it (here NULL itself).
  • error_code(0x0002) — bit-coded: bit 0 = protection violation (0: page not mapped), bit 1 = write, bit 2 = user mode (0: kernel). Reads: the kernel wrote to an unmapped address.
  • [#2] — the die counter; our BUG() was [#1].
  • Tainted: G … D W O — W a WARN fired, O out-of-tree module loaded, D already died once — and the kernel decodes the letters for you on the next line. Always ask what tainted it first.
  • RIP: e0_write.cold+0x16/0x3f [e0_anatomy] — where it died: function+offset/size, and in which module. .cold is the compiler parking an unlikely path in a cold sub-function — real reports are full of .cold / .isra.0 suffixes; they decode the same way.
  • Code: — the faulting instruction in <…>: c7 00 ad de 00 00 is movl $0xdead,(%rax) — and RAX is the NULL it wrote through.
  • Call Trace: — newest first: the write(2) path into us; ? frames are stack leftovers, not confirmed callers.
  • Modules linked in: — every loaded module with its own taint mark; when O is set, look here first.

Mechanism note: on x86 both WARN and BUG() compile to the same ud2 trap; a bug-table entry tells the handler which it was — that's why a BUG() reports invalid opcode. The NULL deref is different: the page-fault handler found no mapping at address 0 and declared an oops.

./TOUCH — decode the RIP yourself

cd /work/E0-anatomy-of-an-oops/module
/work/harness/.build/linux-6.18.33/scripts/faddr2line e0_anatomy.ko 'e0_write.cold+0x16'
# e0_write.cold at /work/E0-anatomy-of-an-oops/module/e0_anatomy.c:53

Line 53 is the *nothing = 0xdead; store — to the line. Use the RIP from your log. One wrinkle worth keeping: the report said +0x16/0x3f, but passing the /0x3f makes faddr2line refuse with a size mismatch — for module symbols the kernel estimates size as the gap to the next symbol; drop the /size and it resolves. The module is built with -g so the .ko carries DWARF; for a crash in the kernel proper, point the same script at vmlinux (scripts/decode_stacktrace.sh does whole traces). The check runs this decode at the end — the decoded: line — re-run it by hand so it's yours.
