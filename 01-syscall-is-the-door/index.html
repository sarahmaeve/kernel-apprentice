$ 01 ./the-syscall-is-the-door

one event · three views

Every interaction between your software and the kernel goes through one narrow door: the syscall. You call a libc function, it traps into the kernel, a specific C function runs, you get an answer back. That door is where almost every "novel software behaves badly on the real box" problem lives.

The challenge: make a specific line of kernel code announce itself in dmesg, firing exactly when — and only when — our test program calls getpid().

There's a small program here, door.c. All it does is ask the kernel for its PID and print it:

pid_t p = getpid();           /* the door: traps into SYSCALL_DEFINE0(getpid) */
printf("door: my pid is %d\n", (int)p);
[01]

./OBSERVE — the tool shows it

check.sh runs door under strace inside the guest. You see the boundary crossing in the output:

getpid()  = 42

strace sits at the door and logs everyone who walks through. That's view one.

[02]

./READ — the code explains it

On the kernel side, getpid() is a tiny function. In the pinned source (harness/.build/linux-6.18.33/), open kernel/sys.c and find:

SYSCALL_DEFINE0(getpid)
{
        return task_tgid_vnr(current);
}

SYSCALL_DEFINE0 defines a syscall taking 0 arguments. current is the running task; task_tgid_vnr returns its PID as seen from the task's namespace. That's the whole door for getpid — view two.

[03]

./TOUCH — your printk confirms it

Reach in. Announce the handler — but only for our process, so we don't drown the log (every program calls getpid; a blind printk here floods — which is exactly why later lessons reach for dynamic tracing instead).

SYSCALL_DEFINE0(getpid)
{
        if (!strcmp(current->comm, "door"))
                pr_info("kernel-apprentice: getpid by %s pid=%d\n",
                        current->comm, task_tgid_vnr(current));
        return task_tgid_vnr(current);
}

Then from the repo root:

make check LESSON=01-syscall-is-the-door

The harness does an incremental rebuild (seconds), boots the guest, runs door under strace, and greps the kernel log. That's view three — proof that exact line ran.

[04]

$ VERIFY

Green only when the serial log contains both:

  • getpid( — the boundary, observed by strace, VIEW 1
  • kernel-apprentice: getpid by door — your line of kernel C, fired VIEW 3

Compare the PID in the strace line with pid= in your printk. Same number. The door, the function behind it, and proof it ran.

[05]

./GRADUATED HINTS

where does getpid live? 
grep -rn "SYSCALL_DEFINE0(getpid)" harness/.build/linux-6.18.33/kernel/
why guard on the process name?

Every process calls getpid constantly. An unconditional printk floods the ring buffer and console. current->comm is the current task's command name ("door"), so guarding on it keeps the log clean.

the exact edit

In kernel/sys.c, replace the body of SYSCALL_DEFINE0(getpid) with the guarded pr_info in the TOUCH section above. strcmp, current, pr_info, and task_tgid_vnr are all already available in that file — no new includes.

If you can find the function behind a syscall and prove it ran, you can do it for any boundary your software crosses. Everything after this is the same move against a bigger door.
