[!]

> this lesson builds a SECOND kernel

KASAN is compile-time instrumentation, so it needs its own kernel built with CONFIG_KASAN — a full second kernel compile (minutes) and ~3 GB of extra disk on the build volume. The base tutorial kernel is untouched, and nothing else in the curriculum depends on this — skip it freely.

This mirrors real life: you build a dedicated KASAN kernel to chase a memory bug, boot it, read the report, then go back to your normal kernel. Once it's built, all three bugs below run on it — no further setup.

KASAN (Kernel Address Sanitizer) inserts a check before every load/store and keeps shadow memory marking which bytes are legal to touch, with poisoned redzones around allocations and a quarantine that keeps freed objects poisoned. An out-of-bounds, use-after-free, or double-free is caught at the offending instruction — not as a mysterious crash far away and later.

[01]

./OBSERVE — three bugs, escalating

Build the overlay kernel once (or let the check do it), then run the lesson:

make kasan-kernel
make check LESSON=C2-kasan-catches-you

The check builds three tiny modules — one bug each — and insmods them on the KASAN kernel. Every unfixed bug prints a full splat and the check stays RED.

./GRADUATED HINTS

Diagnose from the KASAN report first — each module prints its own when you run the check. Open these only as needed; the last one is the answer.

part 1 — out-of-bounds (kasan_oob): how to read its report 
BUG: KASAN: slab-out-of-bounds in kasan_oob_init+0x.../0x... [kasan_oob]
Write of size 1 at addr ffff... by task ...
 <stack of the bad access>          <- the line to look at
Allocated by task ...:
 <stack of the kmalloc>             <- the object you overran
Memory state around the buggy address:
 ... 00 00 fc fc fc ...             <- 00 = in-bounds, fc = redzone
part 2 — use-after-free (kasan_uaf): how to read its report 
BUG: KASAN: slab-use-after-free in kasan_uaf_init+0x.../0x... [kasan_uaf]
Read of size 1 at addr ffff... by task ...
 <stack of the bad access>          <- where you touched freed memory
Allocated by task ...:
 <stack of the kmalloc>
Freed by task ...:                  <- NEW: the kfree that already released it
 <stack of the kfree>

The Freed by task stack pairs the bad read with the exact kfree that released the object — that's how you reason about lifetime bugs.

part 3 — double-free (kasan_df): how to read its report 
BUG: KASAN: double-free in kasan_df_init+0x.../0x... [kasan_df]
Free of addr ffff... by task ...
Allocated by task ...:
 <stack of the kmalloc>
Freed by task ...:                  <- where it was ALREADY freed
 <stack of the first kfree>
the fixes — spoiler, all three
  • Part 1 module/kasan_oob.c — the loop runs i = 0 .. LEN inclusive; stop at LEN - 1: 
    -	for (i = 0; i <= LEN; i++)
+	for (i = 0; i < LEN; i++)
  • Part 2 module/kasan_uaf.c — a lifetime bug, not a typo: the read of c->host happens after kfree(c). Move that read (and its log) before the free.
  • Part 3 module/kasan_df.c — the object is freed twice; free it once (remove the duplicate kfree(p)).

Re-run; with all three fixed, KASAN stays silent and the check goes GREEN.
