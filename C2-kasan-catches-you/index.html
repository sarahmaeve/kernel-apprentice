$ C2 ./kasan-catches-you OPTIONAL

build a KASAN kernel; let it catch a memory bug

[!]

> this lesson builds a SECOND kernel

KASAN is compile-time instrumentation, so it needs its own kernel built with CONFIG_KASAN — a full second kernel compile (minutes) and extra disk on the build volume. The base tutorial kernel is untouched, and nothing else in the curriculum depends on this — skip it freely.

This mirrors real life: you build a dedicated KASAN kernel to chase a memory bug, boot it, read the report, then go back to your normal kernel.

KASAN (Kernel Address Sanitizer) inserts a check before every load/store and keeps shadow memory marking which bytes are legal to touch, with poisoned redzones around allocations. An out-of-bounds or use-after-free is caught at the offending instruction — not as a mysterious crash far away and later.

[01]

./OBSERVE — load it, KASAN catches it

Build the overlay kernel once (or let the check do it), then run the lesson:

make kasan-kernel
make check LESSON=C2-kasan-catches-you

The shipped module/kasan_demo.c writes one byte past a 64-byte kmalloc. On the KASAN kernel that produces a report and the check stays RED, printing the splat for you to read.

[02]

./READ — the report explains it

BUG: KASAN: slab-out-of-bounds in kasan_demo_init+0x.../0x... [kasan_demo]
Write of size 1 at addr ffff... by task ...

 <stack of the bad access>          <- the line to fix

Allocated by task ...:
 <stack of the kmalloc>             <- the object you overran

Memory state around the buggy address:
 ... 00 00 fc fc fc ...             <- 00 = in-bounds, fc = redzone
  • slab-out-of-bounds / Write of size 1 — a one-byte write past a slab object.
  • First stack — the bad access (kasan_demo_init): where the fix goes.
  • Allocated by task — where the memory was kmalloc'd: which object you ran past.
  • Shadow map — 00 is addressable, fc is a redzone. You wrote into the fc.
[03]

./TOUCH — fix the off-by-one

Open module/kasan_demo.c. The loop runs i = 0 .. LEN inclusive, so the last write lands on buf[LEN] — out of bounds. Make it stop at LEN - 1:

-	for (i = 0; i <= LEN; i++)
+	for (i = 0; i < LEN; i++)
 		buf[i] = (char)i;

Re-run; KASAN stays silent and the check goes GREEN.

[04]

$ VERIFY

  • GREEN once the module loads with no KASAN report.
