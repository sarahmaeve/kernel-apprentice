$ 05 ./the-driver-that-oopses

read the crash · find the bug · fix it

A module ships with a bug. Writing to /proc/ka-oops crashes the kernel — an oops (a NULL pointer dereference). The writing process is killed, but the kernel survives — an oops isn't a panic. No TODO markers: you diagnose, like a third-party driver that misbehaves on the real box.

make check LESSON=05-the-driver-that-oopses
Working in the workbench. Edit module/oops_driver.c in this repo (host-editable), then run make check from the host.
[01]

./READ THE OOPS

The check writes to /proc/ka-oops and dumps the trace:

BUG: kernel NULL pointer dereference, address: 0000000000000000
...
RIP: 0010:ka_write+0x.../0x...  [oops_driver]
Call Trace:
 ka_write              [oops_driver]
 proc_reg_write
 vfs_write
 ksys_write

The RIP and the top of the Call Trace name the crashing function — ka_write, in this module. address: 0000…0000 says it dereferenced NULL. Open ka_write: what does it write through, and is that pointer valid?

[02]

./FIND + FIX

ka_write copies the user data into a local buffer (fine), then memcpys it into state->buf. Where is state->buf set? ka_init allocates state but never allocates state->buf — so it's NULL. That's the deref. Give the buffer real memory.

PASS when the write completes with no oops — the log shows kernel-apprentice: ka-oops stored N bytes.

[03]

./GRADUATED HINTS

where does it crash?

The RIP / first Call Trace frame is ka_write; the faulting address: is 0, so it derefs a NULL pointer.

what's NULL?

ka_write memcpys into state->buf. ka_init does state = kzalloc(...) but never sets state->buf — so it stays NULL.

the fix 
state->buf = kmalloc(BUFSZ, GFP_KERNEL);   // in ka_init, after state is allocated
if (!state->buf) {
        kfree(state);
        return -ENOMEM;
}

ka_exit already kfrees state->buf, so no leak.
