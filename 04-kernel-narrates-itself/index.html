$ 04 ./the-kernel-narrates-itself

make a live function speak · no rebuild

In lesson 01 you added a printk and rebuilt to make getpid speak. Here you make the same function speak on an already-running kernel, in milliseconds, with no rebuild at all. That's the dynamic-tracing superpower — how you debug a box you can't recompile.

Every function has an fentry hook (that's ftrace), and you can drop a breakpoint on almost any instruction (that's a kprobe) — all at runtime. The instrumentation is already in the kernel you booted.

Working in the workbench. No module, no kernel edit — you drive the kernel's built-in tracing from userspace (echo/cat into tracefs).
[01]

./OBSERVE — the function tracer narrates

tracefs lives at /sys/kernel/tracing. Narrow the function tracer to one function and watch the kernel report it:

cd /sys/kernel/tracing
echo __do_sys_getpid > set_ftrace_filter   # only narrate this function
echo function       > current_tracer
echo 1 > tracing_on ; ./trigger ; echo 0 > tracing_on
cat trace                                    # -> __do_sys_getpid <-do_syscall_64
[02]

./READ — how it can do that

  • ftrace uses a call __fentry__ the compiler planted at the start of every function (-pg); at runtime the kernel patches those sites to jump into the tracer. (That's the ftrace: allocating … entries line at boot.)
  • kprobes replace an instruction anywhere with a breakpoint, run your handler, then single-step the original. kprobe_events exposes this through tracefs, so one echo registers a probe.

Neither needs the source or a compiler — view two. (SYSCALL_DEFINE0(getpid) from lesson 01 compiles to __do_sys_getpid, the body we trace; the x86 entry __x64_sys_getpid just calls it.)

[03]

./TOUCH — attach a kprobe, live

make check LESSON=04-kernel-narrates-itself

The guest registers a kprobe on the lesson-01 handler and fires it:

cd /sys/kernel/tracing
echo 'p:ka_getpid __do_sys_getpid' > kprobe_events   # define the probe — no rebuild
echo 1 > events/kprobes/ka_getpid/enable
echo 1 > tracing_on ; ./trigger ; echo 0 > tracing_on
cat trace                                              # -> ka_getpid: (__do_sys_getpid+0x0/…)

Compare the effort to lesson 01: there you edited C and waited for a kernel build; here you instrumented the running kernel in one line.

./EXPLORATION — drive it yourself

Capture an argument. kprobe_events can record values — e.g. a probe on do_sys_openat2 that grabs the filename. The syntax is in Further reading.

The friendly frontend. bpftrace (installed in the workbench) is a one-liner wrapper over exactly this machinery. And the big question: when is dropping a printk (lesson 01) the wrong tool and a kprobe the right one? (Hint: production, no source, can't reboot.)
