$ 02 ./printk-and-the-ring-buffer

build the thing that narrates the kernel

dmesg prints kernel messages. But where do they come from, where do they live, and why do some hit the console while others appear only when you ask? The answer is printk and the kernel log ring buffer — and the thing you build here is itself an observability mechanism. Every diagnosis later in this course reads this buffer.

No kernel rebuild this time. A loadable module (.ko) is the gentle, fast loop: compile it against the kernel tree, insmod it, watch it talk, rmmod it.

[01]

./OBSERVE — the tool shows it

Run dmesg -w in one window and load a module in another — lines appear the instant they're emitted. Each carries a log level, 0 (KERN_EMERG) to 7 (KERN_DEBUG).

[02]

./READ — the code explains it

module/hello.c emits exactly one line at each level:

static int __init hello_init(void)
{
        pr_emerg ("kernel-apprentice: hello at EMERG (0)\n");
        /* ... alert, crit, err, warn, notice ... */
        pr_info  ("kernel-apprentice: hello at INFO  (6)\n");
        pr_debug ("kernel-apprentice: hello at DEBUG (7)\n"); /* see exploration */
        return 0;
}
module_init(hello_init);
module_exit(hello_exit);

pr_emerg/pr_info/… are thin wrappers over printk that stamp the level. module_init/module_exit register the load/unload hooks. Read the whole file — it's short, and it's a complete, real kernel module.

[03]

./TOUCH — load it and watch

make check LESSON=02-printk-and-ring-buffer

The harness builds hello.ko against the pinned kernel tree, boots a guest, then the guest's /init:

insmod /hello.ko          # hello_init runs -> 8 printk lines hit the ring buffer
dmesg | grep kernel-apprentice
rmmod hello               # hello_exit runs -> "goodbye"
[04]

$ VERIFY

Green when the serial log contains both:

  • kernel-apprentice: hello at INFO — emitted into the ring buffer LOAD
  • kernel-apprentice: goodbye — the __exit path ran UNLOAD
[05]

./EXPLORATION — drive it yourself

The console filter. dmesg shows all eight… but did all eight reach the serial console as the module loaded? The console only prints messages more urgent than console_loglevel. Lower it:

cat /proc/sys/kernel/printk        # console_loglevel is the first number
echo 1 > /proc/sys/kernel/printk   # now only EMERG/ALERT reach the console

Re-load the module: fewer lines on the console, but dmesg still has them all. The ring buffer holds everything; the console is a filtered view — which is why you dmesg after the fact instead of trusting what scrolled past.

The debug line. pr_debug (level 7) is silent unless dynamic debug is enabled for it. Why might a kernel be built that way? When would you want it on in production?
