$ 03 ./proc-is-code-not-files

the kernel wearing a filesystem costume

cat /proc/self/status prints memory, state, limits — but no file on any disk holds that. /proc is a virtual filesystem: each read calls into kernel code that fabricates the answer right then, so ordinary tools (cat, grep) can read kernel state.

You'll build a /proc entry yourself and prove it's code: read it twice and watch the numbers change.

Working in the workbench. This lesson builds a module from source in the repo (host-editable) and runs make check — no kernel edit needed.
[01]

./OBSERVE — the tool shows it

Read a couple of virtual files twice in a row — the values move because they're computed at read time, not stored:

cat /proc/uptime ; sleep 1 ; cat /proc/uptime   # the numbers advance
cat /proc/self/status | head                      # live state for THIS process
[02]

./READ — the code explains it

module/proc_window.c is a complete module. Its heart is one function:

static int ka_show(struct seq_file *m, void *v)
{
        seq_printf(m, "kernel-apprentice: hello from a seq_file handler\n");
        seq_printf(m, "reading process: %s (pid %d)\n",
                   current->comm, task_tgid_vnr(current));
        seq_printf(m, "jiffies now:     %lu\n", jiffies);
        return 0;
}

ka_show writes the "file contents" with seq_printf every time it's read; the seq_file interface handles buffering and seeking. proc_create("kernel-apprentice", 0444, NULL, &ka_proc_ops) hangs that handler off a name under /proc — view two.

[03]

./TOUCH — build the window, look through it

make check LESSON=03-proc-is-code

The harness builds proc_window.ko, boots a guest, then the guest's /init:

insmod /proc_window.ko          # ka_init -> /proc/kernel-apprentice appears
cat /proc/kernel-apprentice     # your ka_show() runs and produces this text
cat /proc/kernel-apprentice     # read again — jiffies advanced (it's code!)
rmmod proc_window               # ka_exit -> the entry is gone

$ VERIFY

Green when the serial log contains both:

  • kernel-apprentice: hello from a seq_file handler — your handler produced output READ
  • kernel-apprentice: /proc/kernel-apprentice created — the module created the entry CREATE
[05]

./EXPLORATION — drive it yourself

Prove it's code. The check reads the entry twice — compare the two jiffies values in the log. A real file wouldn't change between reads.

Add a field. In make shell, edit module/proc_window.c to seq_printf one more line (e.g. num_online_cpus()), rebuild, reload, read it back. Then look around: /proc/version, /proc/meminfo — each is a *_show function somewhere in the tree.
