cut the firehose down to your signal

A whole-kernel trace is millions of lines a second. B2 has two halves: instrument the code you care about with trace_printk(), and isolate it with per-module / per-PID filters, trace_marker brackets, and tracing_on. B1 followed the path; B2 cuts the noise.

make check LESSON=B2-trace-just-what-matters

module/b2_worker.c exposes /proc/b2-go; writing to it runs three phases — but they run silently, so the check is RED: none of b2: parse, b2: compute, b2: emit appear yet. Runs on the stock kernel.

[01]

./THE CHALLENGE

The three phases run in silence. Make each one write a line to the trace ring buffer (/sys/kernel/tracing/trace) carrying its marker — b2: parse, b2: compute, b2: emit.

trace_printk() is the cheap, hot-path-safe way to write there — printk's cousin, into the trace buffer not the kernel log — and the marker shows up inline with the function trace and your trace_marker notes. The hints below cover the how.

[02]

./ISOLATION TOOLKIT

"The whole kernel, forever" becomes "this module, this process, this moment" — with your trace_printk notes threaded through.

[03]

./GRADUATED HINTS

Hint 1 — what writes to the trace buffer?

printk floods the kernel log; the trace buffer's own printf is trace_printk() — same format args as printk, but it lands in /sys/kernel/tracing/trace. One call per phase. (#include <linux/kernel.h> is already there.)

Hint 2 — what must each line contain?

The check greps the trace for the literal substrings b2: parse, b2: compute, and b2: emit. Each phase's format string must contain its marker.

Hint 3 — the solution 
static noinline void phase_parse(int n)
{
	trace_printk("b2: parse %d bytes\n", n);
}

static noinline void phase_compute(int n)
{
	int sum = n * 3;

	trace_printk("b2: compute sum=%d\n", sum);
}

static noinline void phase_emit(int n)
{
	trace_printk("b2: emit %d bytes\n", n);
}
