$ B2 ./trace-just-what-matters

cut the firehose down to your signal

A whole-kernel trace is millions of lines a second. B2 has two halves: instrument the code you care about with trace_printk(), and isolate it with per-module / per-PID filters, trace_marker brackets, and tracing_on. B1 followed the path; B2 cuts the noise.

make check LESSON=B2-trace-just-what-matters

module/b2_worker.c exposes /proc/b2-go; writing to it runs three phases. Only parse annotates the trace so far, so the check is RED — b2: compute and b2: emit are missing. Runs on the stock kernel.

[01]

./THE CHALLENGE

Add a trace_printk() to phase_compute() and phase_emit() so the trace shows a line containing b2: compute and b2: emit — mirroring the worked example:

trace_printk("b2: parse %d bytes\n", n);   // already there, in phase_parse

trace_printk() writes into the trace ring buffer (/sys/kernel/tracing/trace), not the kernel log — cheap enough to leave in a hot path, and it shows up inline with the function trace and your trace_marker notes.

[02]

./ISOLATION TOOLKIT

The recipe the check runs around your annotations:

toolwhat it does
echo :mod:b2_worker > set_ftrace_filterfunction tracer sees only this module's functions

"The whole kernel, forever" becomes "this module, this process, this moment" — with your trace_printk notes threaded through.

[03]

./GRADUATED HINTS

Hint 1 — where do the annotations go?

In module/b2_worker.c, phase_compute() and phase_emit() each have a TODO(B2) where the trace_printk() belongs. phase_parse() shows the shape.

Hint 2 — what must the line contain?

The check greps the trace for the literal substrings b2: compute and b2: emit. Your format string must contain those — e.g. trace_printk("b2: compute sum=%d\n", n*3);

Hint 3 — the solution 
static noinline void phase_compute(int n)
{
	int sum = n * 3;

	trace_printk("b2: compute sum=%d\n", sum);
}

static noinline void phase_emit(int n)
{
	trace_printk("b2: emit %d bytes\n", n);
}
