you write the kernel code this time

The lessons so far handed you working code to read. Now you write it: a module skeleton + a spec, and two functions to fill in so the kernel counts an event and reports it through /proc. The synthesis of lesson 03 (/proc) and lesson 04 (kprobes) — and your first taste of modification.

The spec — /proc/kernel-apprentice-count, on every read, prints exactly:

getpid calls: <N>

where <N> is the number of times getpid has been called since the module loaded.

Working in the workbench. Edit module/counter.c in this repo (host-editable), then run make check from the host.
[01]

./WHAT'S GIVEN

module/counter.c already does the plumbing — registers a kprobe on __x64_sys_getpid, creates the /proc entry (seq_file, like lesson 03), and keeps an atomic_t ka_count for you. Two functions are stubs:

static int ka_pre_handler(struct kprobe *p, struct pt_regs *regs)
{
        /* TODO 1: bump ka_count by one. (hint: atomic_inc) */
        return 0;
}

static int ka_show(struct seq_file *m, void *v)
{
        /* TODO 2: print "getpid calls: N\n" with atomic_read(&ka_count). */
        return 0;
}

As shipped it compiles and loads, but counts nothing and /proc is empty — so the check is red until you implement both.

./TOUCH

Fill in the two TODOs, then:

make check LESSON=06-instrument-it-yourself

PASS when /proc/kernel-apprentice-count reports a non-zero count after the test calls getpid 50 times. You'll have built a kernel instrument — by hand.
