$ H1 ./step-the-live-kernel

stop time inside the kernel · no rebuild

Lesson 04 let you watch a function run. Here you stop time inside the kernel: break on the very getpid handler from lesson 01, and when the running kernel hits it, freeze — inspect the stack, registers, the next instructions — then let it go.

QEMU is already a debugger. It exposes a gdb stub: launch with -gdb tcp::1234 + -S (start frozen), point gdb vmlinux at it, and you debug the kernel like any program. (Boot with nokaslr so symbols line up.) The thing courses set up laboriously with kgdb, we get almost for free.

Working in the workbench. No module, no kernel edit — you attach gdb to the running guest.
[01]

./TOUCH — break into the running kernel

make check LESSON=H1-gdb-the-live-kernel

The harness boots the guest frozen with a stub, attaches gdb, breaks on __do_sys_getpid, continues until a getpid() hits it, and prints the stack:

Breakpoint 1, __x64_sys_getpid ()    # the getpid handler, frozen mid-call
#0  __x64_sys_getpid ()
#1  do_syscall_64 ()                 # who called it

You break on __do_sys_getpid (the body lesson 04 traced), but gdb labels that aliased address __x64_sys_getpid — same code, different name (kallsyms vs ELF symbols). PASS when gdb stops in the getpid handler on the live kernel: you froze it mid-syscall and looked around — the deepest "reach in" yet.

[02]

$ DO IT YOURSELF

Two shells from make shell — boot frozen in one, drive in the other:

gdb harness/.build/linux-6.18.33/vmlinux
(gdb) target remote :1234
(gdb) break __do_sys_getpid
(gdb) continue
(gdb) backtrace          # the path from userspace into the handler
(gdb) info registers
(gdb) x/4i $pc           # the instructions about to execute
(gdb) finish             # run until this function returns
(gdb) continue

This kernel is built without CONFIG_DEBUG_INFO, so gdb shows function names + offsets, not source lines. Turning on debug info (with the oops/crash module) upgrades every session to full source level — list, info locals, kernel/sys.c:NN.

[03]

./WHY THIS IS THE DIFFERENTIATOR

The SUSE debugging deck spends a whole section on kgdb: a serial console, kgdboc=ttyS0, echo g > /proc/sysrq-trigger to break in. All real, all fiddly. Because our guest is QEMU, we skip every bit of it — the gdb stub is built into the emulator.

That's something a lecture or a video can't easily let you do. When you can't recompile a box but can attach a debugger (a VM, a board with a gdb stub), this is how you find out what the kernel is actually doing — the last resort that works when printk and tracing aren't enough.
