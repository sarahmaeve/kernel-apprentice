be the door this time

You've read the kernel, traced it, instrumented it, fixed it. Now build a piece of it: a real (if tiny) character device — read, write, and an ioctl — so a userspace test passes. The boundary from lesson 01, seen from the kernel side.

module/ka_chardev.c registers /dev/ka-chardev over a small buffer. Make it behave so test.c passes:

  • write(fd, msg, n) → store n bytes, return n
  • ioctl(fd, KA_GET_LEN, &len) → set len to the stored length, return 0
  • read(fd, buf, count) → return the stored bytes (EOF on a second read)
Working in the workbench. Edit module/ka_chardev.c in this repo (host-editable), then run make check from the host.
[01]

./WHAT'S GIVEN

Registration (misc_register), the /dev node, the file_operations table, and the buffer are done. Three handlers are stubs:

static ssize_t ka_read (...) { /* TODO 1 */ return 0;       }
static ssize_t ka_write(...) { /* TODO 2 */ return -EINVAL; }
static long    ka_ioctl(...) { /* TODO 3 */ return -ENOTTY; }

As shipped the module loads but rejects writes and answers no ioctls — so the test fails and the check is red until you implement all three.

[02]

./TOUCH

Implement the three handlers, then:

make check LESSON=07-char-device-with-ioctl

PASS when the test prints TEST PASS — write stored the bytes, the ioctl reported the right length, and read returned them unchanged.
