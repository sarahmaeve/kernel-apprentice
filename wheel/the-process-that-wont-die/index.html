./YOUR JOB

  1. Read the symptom. journaling ticket 1 … and nothing. No error, no exit. kill -9 returns success and changes nothing.
  2. Ask the task's state. grep State /proc/<pid>/status — D (disk sleep), uninterruptible; SigPnd shows the undelivered KILL.
  3. Ask where. cat /proc/<pid>/stack (H2's move): top frame vjournal_write — inside the vendor driver. (The stack file filters the lock plumbing; khungtaskd's dmesg trace shows the __mutex_lock frames.)
  4. Ask who else. dmesg — khungtaskd (D3) reported two blocked tasks, then named the relationship outright: INFO: task ticketd:68 is blocked on a mutex likely owned by task vjournal-compac:67. (Note the 15-char comm truncation.)
  5. Point at the evidence. "ticketd is in uninterruptible sleep inside vjournal_write, waiting on the journal mutex; the holder, vjournal-compact, is itself wedged in D — the lock never releases, so SIGKILL is never delivered."
[03]

./WHY KILL -9 CANNOT WORK

SIGKILL doesn't destroy a process — it's delivered the next time the task runs. A task in TASK_UNINTERRUPTIBLE is not runnable and doesn't process signals; that's the contract (normally held for microseconds around I/O). A task that stays in D is beyond userspace entirely: the only ways out are the awaited resource appearing, or a reboot.

It's also why load climbs while CPUs idle: Linux counts D-state tasks into the load average — the classic signature of this whole incident class.

grep -E '^(State|SigPnd)' /proc/<pid>/status — State: D (disk sleep), and SigPnd: 0000000000000100: that bit is signal 9, the SIGKILL that landed and will never be delivered.

the kernel already told you

dmesg | grep "blocked for more than" — khungtaskd reported both tasks, then named the relationship: blocked on a mutex likely owned by task vjournal-compac:67. Two D-state tasks, one lock: the holder is the root cause; the service is collateral.

[05]

./POST-MORTEM

open after you've driven it — ground truth + the lesson

Ground truth. The vendor driver's compaction thread takes the journal mutex, then waits forever for "device I/O" that never completes (vjournal.c — a stand-in for a dead device or firmware hang). The service's next write queues on that mutex in uninterruptible sleep and becomes unkillable. Nothing is wrong with ticketd — the vendor module wedged while holding a lock the service needs.

Why it's the archetype. "Unkillable process + climbing load + idle CPUs" is a staple page, and the cause is almost always a D-state wait on a dead resource: NFS server gone, dying disk under fsync, wedged vendor driver. The moves are mechanical: state, stack, khungtaskd — who's stuck, where, and who holds what they wait for.

What observability would have caught it faster. khungtaskd had named both tasks before anyone looked; alerting on INFO: task … blocked for more than (or hung_task_panic=1 + kdump in stricter fleets — lesson E1, planned) makes this page self-diagnosing.

check.sh verifies the scenario arms: the service freezes, sits in D, survives SIGKILL, its stack names the module, and khungtaskd reports it. A later facilitator layer grades your evidence.
