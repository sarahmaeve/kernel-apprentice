read the kernel · debug the boundary

Learn to read kernel code and debug how software interacts with the kernel — fast enough to be useful on real problems. The motivating case is concrete: novel software (a service, runtime, driver, appliance) misbehaves on a production-shaped box in ways that never show up on a developer laptop, and you need to localize it.

This is not a path to landing upstream patches or chasing exotic internals. It builds the reader + tracer skill set: fluency at the syscall boundary, a subsystem map to localize a problem, real facility with tracing tools, and the ability to read kernel C and a stack trace.

[01]

./ARCHITECTURE

The container is the workbench. A QEMU VM launched from it is the real, breakable kernel. The safety boundary for kernel experiments is QEMU — not the container.

workbench · container

Pinned kernel source, cross toolchain, QEMU, and tracing tools (strace, perf, bpftrace, trace-cmd). Disposable and reproducible. Shares the host kernel — so it can't boot one you built.

guest · QEMU

The kernel you observe, modify, and break freely. Booted with serial captured for verification. A lean BusyBox initramfs boots in seconds; modules and rebuilds only work on a kernel you control.

[02]

./METHOD — one event, three views

Each lesson anchors on a single observable moment and walks it through three lenses. Symptom-first: you're handed what production shows and you drive.

observe

A tool shows the phenomenon — strace at the door, dmesg on the buffer.

read

Kernel C explains the machinery — the handler behind the syscall, the module behind the printk.

touch

Your own printk / kprobe lets you reach in and confirm that exact line ran.

Why make kernel runs twice. We never guess hashes. The first run shows you the real sha256 so you pin a value you've actually seen — ideally cross-checked against kernel.org — which locks the supply chain for every contributor. Once pinned in harness/versions.env, the build is a single command and the kernel is cached for good.

make help lists every target; make check with no LESSON runs them all.

./WORKBENCH ACCESS

The kernel source lives in a Linux build volume, not on your host filesystem (see harness/ for why), so you read and edit it from inside the workbench.

connect

make shell drops you into the workbench container. VS Code / Cursor users: Attach to Running Container for a full IDE on the volume.

edit

In the shell, edit with nano or vim — e.g. harness/.build/linux-6.18.33/kernel/sys.c. The lesson files in this repo edit normally on your host.

sanity

make doctor checks host prereqs · make validate lints these docs · make status refreshes this dashboard.
