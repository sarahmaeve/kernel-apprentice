what stalled — and in what context

B1 followed the path; B2 cut the noise. B3 is about time and context: the kernel's purpose-built tracers for "longest time with interrupts off?", "which functions are hot?", "how close to a stack overflow?" — and the latency format that decodes the state (irqs off? in a hardirq? preemption disabled?) of every line.

First lesson that needs the Module B tracers. Built your kernel before Module B? The check detects it and tells you to run make kernel. (B1/B2 run on the stock kernel; B3/B4 need the rebuild.)

[01]

./READ — the four views

A — irqsoff latency tracer + the context flags

echo irqsoff > current_tracer records the single longest region the kernel ran with interrupts disabled, and the path that caused it:

# tracer: irqsoff
# latency: 37347 us, #68/68, CPU#0 | (M:PREEMPT(voluntary) ... #P:2)
#  => started at: irqentry_enter
#  => ended at:   irqentry_exit
#                  _-----=> irqs-off/BH-disabled
#                 / _----=> need-resched
#                | / _---=> hardirq/softirq
#                || / _--=> preempt-depth
#  cmd  pid      ||| /  delay  |  caller

That legend is the latency format — the "context" half of the lesson. Every line carries these flags:

flagmeans
dirqs off (interrupts disabled)
h / sin a hardirq / softirq
Nneed-resched — a higher-prio task is waiting
digitpreempt-depth — nested preempt_disable()

Reading the flags tells "slow path with interrupts off" (bad — it delayed every IRQ) from "slow but preemptible" (usually fine). Siblings: swap in preemptoff, preemptirqsoff, or wakeup_rt.

B — function profiler: what's hot

echo 1 > function_profile_enabled counts + times every function; read trace_stat/function<cpu>:

  Function          Hit    Time            Avg
  schedule          923    20703657 us     22430.83 us
  do_idle           866    5806523 us      6704.992 us
  schedule_timeout  645    5398054 us      8369.076 us

Hit = call count, Time = total. (An idle guest is all schedule/do_idle — the scheduler parking the CPU.)

C — stack tracer: how close to the edge

echo 1 > /proc/sys/kernel/stack_tracer_enabled remembers the deepest kernel stack ever seen, frame by frame:

        Depth    Size   Location    (11 entries)
  0)     2712      48   rcu_segcblist_enqueue+0x9/0x50
  4)     2464    1128   mas_wr_bnode+0xb50/0x24c0      <- 1128 bytes in one frame
  6)     1216     768   __mmap_region+0x766/0xe10
  9)      224      48   do_syscall_64+0xb4/0x350

The kernel stack is small and fixed; a fat frame or runaway recursion can overflow it and corrupt memory. The stack tracer finds the worst offender before it bites.

D — snapshot, and dump-on-oops

echo 1 > snapshot atomically swaps the live ring buffer into a snapshot buffer — freeze "the last N events" while tracing keeps running. Its crash-time cousin, echo 1 > /proc/sys/kernel/ftrace_dump_on_oops, dumps the buffer to the console on a panic — the last thing that happened before death, with no disk.

./TOUCH — try the others

T=/sys/kernel/tracing

cat $T/trace_stat/function*                                       # full profiler, all CPUs
[!]

> on the numbers

You're on an emulated (TCG) guest, so absolute times are wildly inflated — a 37 ms "irqs-off" latency would be alarming on real hardware (microseconds is normal). The skill B3 teaches is reading these tools — the worst region, its path, the context flags, the hot functions, the deepest stack — not trusting the emulator's clock.
